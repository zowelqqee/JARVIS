# Документация по проекту JARVIS

## 1. Что это за проект

`JARVIS` в этом репозитории - это локальный помощник для рабочего стола.
Он умеет:

- принимать текстовую команду или голосовую фразу;
- понять, хочет ли пользователь действие или просто ответ на вопрос;
- если это действие, аккуратно разобрать команду, проверить ее, при необходимости спросить уточнение, потом выполнить шаги по одному;
- если это вопрос, ответить отдельно, ничего не выполняя;
- показывать пользователю, что происходит сейчас;
- в части вопросов уметь работать либо по простым локальным правилам, либо через модель OpenAI, но только когда это явно включено.

Самая важная мысль про проект:

- здесь есть **два верхних режима**: `command` и `question`;
- режим ответа на вопрос специально отделен от режима выполнения команд;
- проект старается не делать ничего скрытно;
- если система не уверена, она останавливается и спрашивает;
- если действие опасное, она просит подтверждение;
- если вопрос нельзя честно ответить, она не должна выдумывать ответ.

Это не "всё умеющий агент". Это скорее аккуратный, ограниченный и проверяемый помощник.

## 2. Как смотреть на проект правильно

Если читать этот репозиторий как лекцию, то лучше держать в голове такую простую схему:

1. Пользователь что-то вводит в `cli.py`.
2. `interaction/interaction_manager.py` решает, куда это отправить.
3. `interaction/interaction_router.py` выбирает: команда, вопрос или нужно уточнение.
4. Дальше идет либо ветка команд, либо ветка ответов.

То есть проект не "хаотичный". Он довольно строго разложен по слоям:

- ввод;
- выбор режима;
- разбор команды;
- проверка команды;
- уточнение;
- план шагов;
- выполнение шагов;
- показ результата;
- отдельная ветка ответов на вопросы;
- отдельные файлы для проверок, релизных отчетов и голоса.

## 3. Что главное понять за первые 15 минут

Если совсем коротко, то весь проект можно объяснить так:

- `cli.py` - вход в программу;
- `InteractionManager` - главный диспетчер;
- `InteractionRouter` - решает, это команда или вопрос;
- `RuntimeManager` - весь жизненный путь команды;
- `parse_command()` - превращает фразу в структуру `Command`;
- `validate_command()` - проверяет, можно ли такую команду выполнять;
- `build_execution_plan()` - делает список шагов;
- `execute_step()` - выполняет один шаг на macOS;
- `answer_question()` - отдельный путь для вопросов;
- `map_visibility()` и `interaction_output_lines()` - превращают внутреннее состояние в понятный текст для пользователя.

Если эти точки понятны, то дальше весь проект уже разбирается спокойно.

## 4. Как запустить проект

Базовый запуск:

```bash
python3 cli.py
```

После этого открывается простой текстовый режим.
Там можно писать обычные фразы вроде:

- `open Safari`
- `open Telegram and Safari`
- `what can you do?`
- `why are you waiting?`

Команды оболочки внутри `cli.py`:

- `help`
- `voice`
- `speak on`
- `speak off`
- `reset`
- `quit`

Есть и служебные команды для проверки качества режима вопросов:

- `qa backend`
- `qa model`
- `qa smoke`
- `qa gate`
- `qa gate strict`
- `qa beta`
- `qa checklist`
- `qa release review`
- `qa readiness`

Важно:

- реальное выполнение рабочего стола сделано под macOS;
- голосовой ввод и озвучка тоже заточены под macOS;
- часть режима ответов через модель включается только через переменные окружения.

## 5. Главная идея проекта на одном примере

Пример 1. Пользователь пишет:

```text
open Telegram and Safari
```

Что происходит:

1. `cli.py` принимает строку.
2. `InteractionManager.handle_input()` запускает верхний разбор.
3. `route_interaction()` понимает, что это команда.
4. `RuntimeManager.handle_input()` запускает ветку команды.
5. `normalize_input()` чистит строку.
6. `parse_command()` строит `Command` с намерением `open_app`.
7. `validate_command()` проверяет, что цели понятны.
8. `build_execution_plan()` делает 2 шага `open_app`.
9. `execute_step()` выполняет шаги по одному.
10. `map_visibility()` формирует видимое состояние.
11. `interaction_output_lines()` печатает итог.

Пример 2. Пользователь пишет:

```text
what can you do?
```

Что происходит:

1. `cli.py` принимает строку.
2. `InteractionManager.handle_input()` запускает верхний разбор.
3. `route_interaction()` понимает, что это вопрос.
4. `answer_question()` запускает ветку ответов.
5. `classify_question()` относит вопрос к типу `capabilities`.
6. `build_grounding_bundle()` собирает допустимые источники.
7. Выбирается движок ответа.
8. Формируется `AnswerResult`.
9. `map_interaction_visibility()` делает видимый ответ.
10. `interaction_output_lines()` печатает итог.

Ключевой момент:

- вопрос не превращается в команду;
- команда не должна молча стать ответом;
- смешанный запрос сначала требует уточнения.

## 6. Два режима работы

### Режим `command`

Это путь, где JARVIS что-то делает:

- открывает приложение;
- открывает файл;
- открывает папку;
- открывает сайт;
- показывает окна;
- ищет локальные файлы;
- собирает короткое рабочее окружение;
- закрывает приложение только через подтверждение;
- может остановиться и спросить уточнение.

### Режим `question`

Это путь, где JARVIS ничего не выполняет, а только отвечает.

Сейчас он умеет отвечать, например, на такие вещи:

- что он умеет;
- что делает сейчас;
- почему он остановился;
- где лежит нужный файл в проекте;
- как устроены правила работы;
- что было недавно сделано;
- из какого источника взят ответ.

Отдельно есть более широкий путь через модель OpenAI.
Но он:

- не должен ломать безопасные правила;
- не должен притворяться локальной документацией;
- включается только через настройки;
- обвязан проверками и отчетами.

## 8. Структура репозитория

Ниже карта папок простым языком.

### Корень проекта

- `cli.py`
  Главный вход. Поднимает `RuntimeManager`, `InteractionManager`, `SessionContext`, голосовой вывод и крутит цикл `jarvis>`.
  Файл большой. Не пугайся: в нем много служебных команд для QA и отчетов, поэтому он длиннее, чем основной поток программы.

- `voiceover_plan.md`
  Похоже на рабочую заметку про голосовую часть. Это не основной код выполнения.

- `tmp/qa/`
  Здесь лежат служебные артефакты проверок. Это результаты проверок, а не логика продукта.

- `documentation.md`
  Этот файл.

- `architecture.md`
  Файл со схемами и связями между слоями.

### `interaction/`

- `interaction/interaction_manager.py`
  Верхний координатор.
  Он не выполняет команды сам, а направляет запрос в нужную ветку:
  либо в `RuntimeManager`, либо в `answer_question()`.
  Также он хранит контекст недавнего ответа, чтобы потом понимать фразы вроде `Explain more`.

- `interaction/interaction_router.py`
  Развилка "команда или вопрос".
  Здесь живут правила приоритета:
  если система ждет подтверждение, ответы вроде `yes` и `no` важнее всего;
  если вопрос смешан с командой, вместо молчаливого выбора будет уточнение.

### `input/`

- `input/adapter.py`
  Самая базовая чистка текста.
  Убирает лишние пробелы и отбрасывает пустой или бессмысленный ввод.

- `input/voice_input.py`
  Реальный одноразовый захват голоса на macOS.
  Этот файл умеет:
  компилировать небольшой нативный помощник,
  запускать его,
  получать распознанный текст,
  обрабатывать типовые ошибки доступа к микрофону и распознаванию речи.

- `input/voice_normalization.py`
  Чистит распознанную речь.
  Убирает `jarvis`, приветствия, повторы, шумные слова, а еще приводит некоторые русские команды к тому виду, который понимает парсер.

- `input/macos_voice_capture.m`
  Нативный код помощника для macOS, который занимается самим распознаванием речи.

- `input/macos_voice_capture_info.plist`
  Описание прав и упаковки для этого помощника.

### `voice/`

- `voice/asr_service.py`
  Обертка над захватом голоса.
  Делает две вещи подряд:
  берет распознанный текст и нормализует его.

- `voice/session.py`
  Описывает один голосовой проход как маленький жизненный цикл:
  `idle -> listening -> transcribing -> routing`.

- `voice/language.py`
  Очень простое определение языка:
  если есть кириллица, значит `ru-RU`, иначе `en-US`.

- `voice/speech_presenter.py`
  Превращает результат работы JARVIS в короткую фразу для озвучки.
  Здесь много правил для русского и английского текста.
  Это не выполнение логики, а только "как сказать ответ вслух".

- `voice/tts_provider.py`
  Общий договор для озвучки.

- `voice/tts_macos.py`
  Реальная озвучка через macOS-команду `say`.

### `runtime/`

- `runtime/runtime_manager.py`
  Самый важный файл командной ветки.
  Он:
  принимает новый ввод,
  умеет продолжать зависшую команду,
  запускает разбор,
  запускает проверку,
  строит план,
  выполняет шаги,
  блокируется на уточнении,
  блокируется на подтверждении,
  обновляет `SessionContext`,
  строит результат `RuntimeResult`.

- `runtime/state_machine.py`
  Фиксирует допустимые переходы состояний.
  Это защита от хаотичных переходов.

### `parser/`

- `parser/command_parser.py`
  Большой файл разбора команд.
  Он пытается превратить человеческую фразу в `Command`.
  Поддерживает:
  открытие приложений;
  открытие файлов и папок;
  открытие сайтов;
  показ окон;
  локальный поиск;
  короткую настройку рабочего окружения;
  ответы на подтверждение;
  команды-продолжения вроде `now open browser too`;
  выбор результата поиска вроде `open the first result`.

  Здесь же живут:

  - алиасы приложений, например `browser -> Safari`, `code -> Visual Studio Code`;
  - алиасы сайтов вроде `github -> https://github.com`;
  - ссылки на недавний контекст вроде `this folder`, `same project`, `that file`;
  - перевод некоторых русских слов в понятные парсеру варианты.

### `validator/`

- `validator/command_validator.py`
  Проверяет, имеет ли разобранная команда право жить дальше.
  Основные задачи:

  - порог уверенности;
  - правильный тип цели;
  - есть ли нужные параметры;
  - понятна ли ссылка на прошлый контекст;
  - нет ли неоднозначности.

  Очень полезный файл, если хочется понять, какие команды проект реально считает корректными.

### `clarification/`

- `clarification/clarification_handler.py`
  Создает короткие уточняющие вопросы.
  Потом умеет применить ответ пользователя обратно к `Command`.

  Пример:

  - было `open notes`;
  - выяснилось, что целей несколько;
  - файл строит фразу вроде "какой именно?";
  - затем вносит ответ пользователя обратно в команду.

### `context/`

- `context/session_context.py`
  Короткая память текущей сессии.
  Важная мысль: это не долговременная память.

  Тут хранится:

  - активная команда;
  - шаг и состояние выполнения;
  - недавние цели;
  - недавняя папка проекта;
  - недавний основной объект;
  - результаты последнего поиска;
  - недавний ответ на вопрос и его источники;
  - временное уточнение для смешанного ввода.

### `planner/`

- `planner/execution_planner.py`
  Получает уже проверенную команду и строит список шагов `Step`.
  Именно здесь фраза превращается в последовательность действий.

  Примеры:

  - `open Telegram and Safari` -> два шага `open_app`;
  - `open Downloads folder` -> один шаг `open_folder`;
  - `open the newest md file in this folder` -> сначала `search_local`, потом `open_file`;
  - `set up workspace for JARVIS` -> несколько шагов для подготовки окружения.

  Также здесь строятся точки подтверждения перед опасными действиями.

### `executor/`

- `executor/desktop_actions.py`
  Просто список допустимых действий рабочего стола.

- `executor/desktop_executor.py`
  Реальное исполнение на macOS.
  Что умеет по факту:

  - открыть приложение;
  - открыть файл;
  - открыть папку;
  - открыть сайт;
  - показать список окон;
  - закрыть приложение;
  - сделать локальный поиск.

  Что важно:

  - используется `open` для открытия;
  - используется `osascript` для некоторых действий с приложениями;
  - для списка окон используется CoreGraphics через `ctypes`;
  - исполнитель честно падает там, где функция ненадежна или не доделана.

  Важное ограничение:

  - `focus_window` и `close_window` сейчас специально оставлены как явно неготовые пути;
  - то есть модель данных их знает, но реальный исполнитель их не обещает делать надежно.

### `confirmation/`

- `confirmation/confirmation_gate.py`
  Строит текст подтверждения для команды или отдельного шага.

### `ui/`

- `ui/visibility_mapper.py`
  Один из самых полезных файлов для понимания "что видит пользователь".
  Он берет внутреннее состояние и делает удобный словарь:

  - текущее состояние;
  - краткое описание команды;
  - текущий шаг;
  - завершенные шаги;
  - причина блокировки;
  - сообщение об ошибке;
  - результат;
  - краткие результаты поиска;
  - краткие результаты списка окон;
  - данные по ответу на вопрос.

- `ui/interaction_presenter.py`
  Делает уже готовые строки для консоли.
  То есть `visibility_mapper.py` готовит смысл, а этот файл готовит видимый текст.

### `types/`

Это общий словарь структур данных проекта.
Их очень полезно читать рано, потому что они объясняют форму всего проекта.

- `types/command.py`
  `Command` и список намерений `IntentType`.

- `types/target.py`
  `Target` и типы целей: приложение, файл, папка, окно, браузер, неизвестно.

- `types/step.py`
  `Step`, действия и состояния шага.

- `types/runtime_state.py`
  Список состояний командной ветки.

- `types/validation_result.py`
  Результат проверки команды.

- `types/planned_command.py`
  Готовый план команды.

- `types/action_result.py`
  Результат выполнения шага.

- `types/clarification_request.py`
  Структура уточняющего вопроса.

- `types/confirmation_request.py`
  Структура запроса подтверждения.

- `types/question_request.py`
  Структура вопроса в ветке ответов.

- `types/answer_result.py`
  Структура ответа на вопрос.

- `types/interaction_kind.py`
  Верхние режимы: `command`, `question`, `clarification`.

- `types/interaction_result.py`
  Общий результат одного пользовательского ввода.

- `types/jarvis_error.py`
  Общая структура ошибок.

### `qa/`

Папка `qa/` очень важная, но в ней смешаны **две большие роли**:

1. обычная логика ответа на вопросы;
2. служебная логика проверки и раскатки модельного режима.

Поэтому удобнее делить ее на блоки.

#### `qa/` - ядро ветки ответов

- `qa/answer_backend.py`
  Общий договор для движков ответов.

- `qa/answer_config.py`
  Загружает настройки из переменных окружения.
  Здесь решается:

  - какой движок ответа включен;
  - какая модель используется;
  - включен ли широкий режим через модель;
  - включен ли запасной откат на простой ответ;
  - какая стадия раскатки выбрана.

- `qa/answer_engine.py`
  Главный вход в ветку ответов.
  Он:

  - классифицирует вопрос;
  - собирает допустимые источники;
  - выбирает движок ответа;
  - запускает его;
  - проверяет атрибуцию источников.

- `qa/capability_catalog.py`
  Небольшой каталог того, что система считает поддерживаемыми возможностями.

- `qa/deterministic_backend.py`
  Простой предсказуемый движок ответов.
  Он отвечает по шаблонам и правилам, без внешней модели.

- `qa/source_selector.py`
  Выбирает, какие источники можно использовать для конкретного вопроса.

- `qa/source_registry.py`
  Реестр источников.
  Здесь явно записано: для какого типа вопроса какие файлы и какие опорные утверждения разрешены.

- `qa/grounding.py`
  Собирает "пакет опоры" для ответа:

  - список допустимых файлов;

## 9. Главные структуры данных

Проект очень удобно понимать через его структуры.

### `Command`

Файл: `types/command.py`

Это основная структура командной ветки.
Она хранит:

- исходный текст;
- намерение `intent`;
- список целей `targets`;
- параметры;
- уверенность;
- нужен ли запрос подтверждения;
- список шагов выполнения;
- короткое сообщение о статусе.

Грубо говоря, это "понятая команда".

### `Target`

Файл: `types/target.py`

Это объект, над которым надо что-то сделать.
Типы цели:

- приложение;
- файл;
- папка;
- окно;
- браузер;
- неизвестная цель.

### `Step`

Файл: `types/step.py`

Это один исполнимый шаг.
Например:

- открыть приложение;
- открыть файл;
- сделать поиск;
- закрыть приложение.

Команда может состоять из одного шага или из нескольких шагов.

### `QuestionRequest`

Файл: `types/question_request.py`

Это разобранный вопрос.
Там хранится:

- исходный текст;
- тип вопроса;
- область ответа;
- дополнительные ссылки на контекст;
- уверенность;
- требуется ли локальная опора на источники.

### `AnswerResult`

Файл: `types/answer_result.py`

Это готовый ответ на вопрос.
Там есть:

- текст ответа;
- источники;
- объяснение, какой источник что подтверждает;
- предупреждение;
- вид ответа;
- происхождение ответа.

### `InteractionResult`

Файл: `types/interaction_result.py`

Это общий контейнер результата одного пользовательского ввода.
Он может содержать:

- результат команды;
- результат ответа;
- уточнение;
- ошибку;
- видимое представление для консоли.

### `JarvisError`

Файл: `types/jarvis_error.py`

Это общая структура ошибки по всему проекту.
Хороший файл для быстрого понимания, какие виды проблем вообще предусмотрены.

## 10. Как проходит команда по системе

### Шаг 1. Очистка ввода

`input/adapter.py` приводит ввод к нормальному виду.

### Шаг 2. Выбор режима

`interaction/interaction_router.py` решает:

- это команда;
- это вопрос;
- это смешанный запрос, который надо уточнить.

### Шаг 3. Разбор в `Command`

`parser/command_parser.py` превращает фразу в структуру.

### Шаг 4. Проверка

`validator/command_validator.py` проверяет:

- хватает ли уверенности;
- понятна ли цель;
- нет ли неоднозначности;
- не отсутствует ли обязательный параметр.

### Шаг 5. Уточнение, если надо

Если чего-то не хватает, `clarification/clarification_handler.py` делает один короткий вопрос.

### Шаг 6. План

`planner/execution_planner.py` строит список `Step`.

### Шаг 7. Подтверждение, если действие опасное

`confirmation/confirmation_gate.py` строит запрос подтверждения.

### Шаг 8. Выполнение

`executor/desktop_executor.py` выполняет шаги на macOS.

### Шаг 9. Показ результата

`ui/visibility_mapper.py` и `ui/interaction_presenter.py` формируют текст для пользователя.

### Шаг 10. Обновление контекста

`context/session_context.py` сохраняет короткий контекст:

- последнюю цель;
- последнюю папку;
- результаты поиска;
- состояние подтверждения;
- состояние выполнения.

## 11. Как проходит вопрос по системе

### Шаг 1. Выбор режима

`InteractionRouter` решает, что это вопрос.

### Шаг 2. Классификация вопроса

`qa/answer_engine.py` через `classify_question()` определяет тип вопроса.

### Шаг 3. Выбор допустимых источников

`qa/source_selector.py` и `qa/source_registry.py` выбирают, на что вообще можно опираться.

### Шаг 4. Сбор пакета опоры

`qa/grounding.py` добавляет:

- разрешенные файлы;
- видимое текущее состояние;
- недавний контекст;
- заметки по правилам ответа.

### Шаг 5. Выбор движка ответа

Есть два варианта:

- `qa/deterministic_backend.py`
- `qa/llm_backend.py`

### Шаг 6. Проверка результата

`qa/grounding_verifier.py` следит, чтобы ответ был честным.

### Шаг 7. Показ пользователю

Ответ тоже проходит через `ui/visibility_mapper.py` и `ui/interaction_presenter.py`.

## 14. Что проект специально не делает

Это очень важный раздел, потому что ограничения здесь не случайные.

Проект специально:

- не исполняет фоновые задачи;
- не ведет длинные автономные сценарии;
- не хранит долговременную память о пользователе;
- не выполняет вопрос как действие;
- не делает скрытых повторных попыток;
- не делает опасные действия без подтверждения;
- не притворяется, что что-то выполнил, если не выполнил;
- не должен подсовывать выдуманные источники;
- по умолчанию не является интернет-поиском общего назначения.

## 15. Главные состояния командной ветки

Состояния лежат в `types/runtime_state.py`, а допустимые переходы - в `runtime/state_machine.py`.

Список состояний:

- `idle`
- `parsing`
- `validating`
- `awaiting_clarification`
- `planning`
- `executing`
- `awaiting_confirmation`
- `completed`
- `failed`
- `cancelled`

Очень полезная мысль:

- состояние вопросной ветки не притворяется командным состоянием;
- вопрос отвечает отдельно;
- командная машина состояний живет только для команд.

## 17. Как устроены служебные проверки качества

Здесь есть отдельный большой пласт, который нужен не для пользователя, а для качества.

Основная идея:

- есть набор сценариев в `evals/qa_cases.json`;
- `evals/run_qa_eval.py` умеет прогонять их пакетно;
- можно сравнивать простой и модельный путь;
- можно смотреть стабильность нескольких прогонов;
- можно хранить артефакты проверок в `tmp/qa/`;
- для бета-режима есть ручной чек-лист и релизный обзор.

То есть проект не просто "умеет отвечать", а еще пытается проверять себя формально.

## 18. На что обратить внимание при чтении `cli.py`

`cli.py` длинный, но по смыслу в нем три большие части:

1. обычный цикл ввода пользователя;
2. голосовые команды и озвучка;
3. длинный набор вспомогательных команд для QA и релизных артефактов.

Если читаешь впервые, не пытайся понимать весь файл подряд.
Лучше сначала прочитать:

- `main()`
- `_handle_cli_command()`
- `_dispatch_runtime_input()`
- `_build_default_interaction_manager()`

И только потом уже смотреть все функции `qa_*`.

## 19. Если нужно добавить новую команду

Простой маршрут изменений обычно такой:

1. Добавить или уточнить намерение в `types/command.py`, если это новый тип команды.
2. Научить `parser/command_parser.py` распознавать новую фразу.
3. Добавить проверку в `validator/command_validator.py`.
4. Добавить шаги в `planner/execution_planner.py`.
5. Добавить действие в `executor/desktop_actions.py`, если нужен новый тип шага.
6. Реализовать действие в `executor/desktop_executor.py`.
7. Если нужно подтверждение, учесть это в планировщике и `confirmation/confirmation_gate.py`.
8. Обновить видимое поведение при необходимости в `ui/visibility_mapper.py`.
9. Добавить тесты.

Хорошая новость:

- структура довольно честная;
- новый путь обычно приходится проводить через несколько файлов;
- зато из-за этого код легко проверять.

## 20. Если нужно добавить новый тип вопроса

Обычно порядок такой:

1. Добавить тип в `types/question_request.py`.
2. Научить `qa/answer_engine.py` распознавать его.
3. Добавить источники в `qa/source_registry.py`.
4. При необходимости уточнить выбор темы в `qa/source_selector.py`.
5. Добавить ответ в `qa/deterministic_backend.py`.
6. Если вопрос должен работать и через модель, проверить схему и разбор в OpenAI-пути.
7. Добавить тесты.

## 22. Что полезно запомнить напоследок

Самая точная короткая формула проекта такая:

- `cli.py` принимает ввод;
- `interaction/` выбирает путь;
- `runtime/` ведет команды;
- `qa/` ведет ответы;
- `planner/` строит шаги;
- `executor/` делает реальные действия;
- `context/` держит короткую память;
- `ui/` показывает правду о состоянии;
- `voice/` добавляет голосовой ввод и озвучку;
